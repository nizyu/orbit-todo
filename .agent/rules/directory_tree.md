# Directory Tree Rules

今後の拡張やコンポーネントの再利用性を高めるため、以下のディレクトリ構成とコンポーネント分離のルールを遵守してください。

## アプリケーション層 (`app/` ディレクトリ内)

- **`app/routes/`**:
  - ルーティング定義およびページのエントリーポイントとなるコンポーネントを配置します。
  - 基本的にルーティングの定義と、`<AuthGuard>` 等のラッパー要素を通じたメインコンポーネントの呼び出しのみを行い、**具体的なレンダリングUIや状態管理ロジックはここには記述しません。**

- **`app/components/`**:
  - ページや機能（ドメイン）ごとにディレクトリを分けてコンポーネントを配置します。
  - ルーティングから直接呼び出される「画面単位のコンテンツコンポーネント」（例: `TasksContent.tsx`）から、それを構成する個別コンポーネントまで、機能ごとに関心事をまとめます。
  - **ディレクトリ構成例**:
    - `components/tasks/`: タスク（ToDo）機能に関連するコンポーネント群（`TasksContent.tsx`, `TaskList.tsx`, `TaskForm.tsx` など）
    - `components/categories/`: カテゴリ管理機能に関連するコンポーネント群
    - `components/history/`: 履歴機能に関連するコンポーネント群
  - アプリ全体で共通して使われるUI要素（ヘッダー、ボタンなど）は、`components/` 直下、もしくは将来的に `components/ui/` 等の汎用ディレクトリへ配置します。

- **`app/lib/` もしくは `app/utils/`** (必要に応じて作成):
  - 共通のユーティリティ関数やFirebaseなどの外部サービス設定ファイル（`firestore.ts`, `firebase-auth.tsx`）を配置します。

- **`app/types/`** (必要に応じて作成):
  - 複数のコンポーネントやルート間で使い回す共通のTypeScript型定義を配置します。

## 実装におけるルール
1. **画面（機能）ごとのディレクトリ分割**: 新しい機能や画面を追加する際は、必ず `components/` 配下にその機能名を冠した専用ディレクトリ（例: `components/settings/`）を作成し、その中にコンポーネント群を格納してください。
2. **ルートファイルの薄さの維持**: `routes/` 側のファイルが肥大化したり、状態(state)やフックを持つことがないようにしてください。
3. **命名規則の統一**: 基本的なドメインモデルとして「タスク」を扱う場合、ファイル名・コンポーネント名ともに `Task` に統一すること（`Todo` や `ToDo` は使わず `Task` とする）。
